# ServiceNow — Cognigy Knowledge Connector

A Cognigy.AI Knowledge Connector that synchronizes **ServiceNow Knowledge Base articles** into a Cognigy Knowledge Store. Runs on Cognigy's built-in Knowledge Connector sync schedule (daily by default in 2026.5+; time selectable in 2026.7+; manual "Sync now" available at any time).

## What it ingests

- Every `kb_knowledge` article in workflow state `published` (configurable) on the connected ServiceNow instance.
- One **Knowledge Source per article**, keyed by the article `sys_id` so renames don't fork duplicates and retired articles get deleted on the next sync.
- Article `text` (HTML body) is stripped to plain text and split into ~1,200-character chunks with overlap. Each chunk carries metadata: `sysId`, `number`, `title`, `knowledgeBase`, `language`, `workflowState`, `lastModified`, `chunkIndex`, `chunkCount`.

Not in v1: attachments, KB embedded images, draft/retired articles (filterable in), per-user ACL enforcement.

## ServiceNow setup (one-time)

You need an Inbound OAuth Application Registry entry on the ServiceNow instance and a service-account user.

1. In ServiceNow, navigate to **System OAuth → Application Registry**.
2. Click **New** → **Create an OAuth API endpoint for external clients**.
3. Fill in:
   - **Name**: `Cognigy Knowledge Connector`
   - **Client ID** is generated for you
   - **Client Secret**: ServiceNow generates this on save — **copy it immediately**, you can't see it again
4. Save. The token endpoint is `https://<instance>.service-now.com/oauth_token.do`.
5. Create or pick a service-account user (e.g. `svc_servicenow_poc`) with:
   - the `knowledge` role (or read ACL on `kb_knowledge` and the target Knowledge Bases),
   - **MFA disabled** (the password grant cannot satisfy MFA prompts),
   - active, not locked, password not flagged for reset.

## Build

```bash
npm install
npm run build
```

Produces `service-now-knowledge.tar.gz`. (You also need `icon.png` — 64×64 — in the project root for the build to succeed; drop one in before running `npm run build`.)

## Install into Cognigy.AI

1. **Manage → Extensions → Upload Extension**, upload the `.tar.gz`.
2. **Build → Knowledge** → open or create a Knowledge Store.
3. **+ Add Knowledge** → pick **ServiceNow** from the Type list.
4. Fill in the fields (below) and save. Cognigy will trigger an initial ingest and from then on run on the store's schedule.

## Configuration

Credentials live on a Cognigy **Connection** (encrypted, reusable). Create it once via **Manage → Connections** or inline from the Knowledge Source form.

### ServiceNow Connection fields

| Field | Notes |
|---|---|
| `instance` | Base URL, e.g. `https://aafesdev.service-now.com` |
| `clientId` | Client ID from the Application Registry record |
| `clientSecret` | The secret value generated by ServiceNow on save |
| `username` | Service-account username with `knowledge` role |
| `password` | Service-account password |

The connector uses the **OAuth 2.0 Password grant** (`grant_type=password`). All five fields above are sent in the body of the token request to `/oauth_token.do`. Mark `clientSecret` and `password` as **secret** when creating the Connection in Cognigy so they're stored encrypted.

### Knowledge Source fields (per instance of the connector)

| Field | Required | Default | Notes |
|---|---|---|---|
| ServiceNow connection | ✅ | — | Pick the connection above |
| Knowledge Bases | — | (all) | Comma-separated `kb_knowledge_base` sys_ids. Blank = pull every KB on the instance. |
| Article numbers | — | (all) | Comma-separated KB article numbers (`KB0010001, KB0017559`). Blank = no number filter. ANDed with the other filters. |
| Language filter | — | (user default) | Two-letter language code (`en`, `es`, …) or comma-separated list. Blank = service-account user's preferred language. |
| Workflow state | — | `published` | The `workflow_state` value to filter on. Clear to ingest drafts/retired too. |
| Page size | — | `200` | Articles fetched per KM API request. |

## Sync semantics

- The connector calls `api.upsertKnowledgeSource` per article with `externalIdentifier = sys_id` and `contentHashOrTimestamp = sys_updated_on`. Cognigy uses these to skip articles whose `sys_updated_on` hasn't changed — only modified articles are re-chunked.
- After the loop, any Knowledge Source previously created by this connector whose `externalIdentifier` is no longer present in the ServiceNow result set is deleted via `api.deleteKnowledgeSource`. That handles retire/delete in ServiceNow.
- Empty articles (no body after HTML stripping) are skipped quietly without disturbing previously-ingested copies.

## API endpoints used

- `POST {instance}/oauth_token.do` — `grant_type=password` with all five params in the body (clientId, clientSecret, username, password, grant_type). Returns access token, cached for the run.
- `GET {instance}/api/sn_km_api/knowledge/articles` — the **scoped** Knowledge Management API. Paginated via `limit` + `offset`, body and metadata fetched via `fields=text,sys_updated_on,kb_knowledge_base,workflow_state,language` so each page returns full article content in one call. Filtered via `filter=workflow_state=published` and optionally `kb=<sys_id_csv>` and `language=<code>`.

The connector deliberately uses the scoped KM API rather than the global `/api/now/table/kb_knowledge` Table API because most ServiceNow OAuth registry records are restricted to scoped APIs only — the Table API returns 403 `"Access to unscoped api is not allowed"` against such clients.

## Troubleshooting

- **401 from `/oauth_token.do`** — wrong client_id/secret, or `glide.oauth.inbound.client.credential.grant_type.enabled` is `false` on the instance, or no OAuth Application User is set on the registry record.
- **403 from `/api/now/table/kb_knowledge`** — the OAuth Application User lacks read on `kb_knowledge` or on the specific Knowledge Base. Add the `knowledge` role.
- **Articles missing** — check the `Workflow state` filter (default `published`), and the `Knowledge Bases` filter if you set one.
- **HTML noise in chunks** — content is run through `html-to-text` (`a` ignored, `img/script/style` skipped). For exotic embedded structures consider extending `extractors.ts`.
- **Rate limiting** — the HTTP client retries on 429/503/504 with `Retry-After` honoured up to 5 attempts.
